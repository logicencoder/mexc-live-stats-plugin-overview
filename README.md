# MEXC Live Stats

![MEXC Live Stats — public trading statistics dashboard](assets/live-dashboard.png)

**Live MEXC spot statistics** on Logic Encoder — trade tape, per-pair analytics, coin search, CSV export, and indexable pages for individual USDT markets. Traders and researchers open [logicencoder.com/mexc-app/](https://logicencoder.com/mexc-app/) for a full-market grid with live prices and volume; each pair also gets its own SEO URL with charts, bot-activity panels, and structured data for search engines. The product is built for anyone who wants MEXC-wide visibility without running their own ingest stack — the backend subscribes to the exchange protobuf feed once and fans out to the public site, per-coin SSR pages, and operator dashboards.

## Tech stack

| Layer | Technologies |
|-------|--------------|
| WordPress plugin | PHP, shortcodes, wp-admin dashboards, WordPress AJAX, WordPress REST ingest |
| Public UI | HTML, CSS, vanilla JavaScript, MessagePack WebSocket client |
| Live backend | Python 3, FastAPI, MEXC protobuf WebSocket ingest, PostgreSQL |
| SEO SSR | Node.js Express — per-symbol HTML + Schema.org JSON-LD |
| Snapshots | Static HTML/JSON + chart PNGs pushed from backend |
| Search | XML sitemap for `/mexc/{SYMBOL}/`, IndexNow queue, auto-push on publish |
| Data | PostgreSQL trade history on backend; WordPress options for coin lists and SEO state |
| Hosting | WordPress on shared hosting; Python/Node services on self-hosted Linux servers |

## Live dashboard

The main app at **[logicencoder.com/mexc-app/](https://logicencoder.com/mexc-app/)** is a full-screen market browser. Shortcode **`[mexc_dashboard]`** drops the same shell into any WordPress page; add `symbol="PIUSDT"` to focus one pair in an embed.

Visitors get:

- **Coin grid** — searchable list of 1,400+ USDT pairs with live last price, direction, and trades-per-minute hints. Display modes switch between ticker-only, full name, or both so dense layouts stay readable on mobile.
- **Live trade tape** — scrollable deal stream per selected pair; freeze the panel when you want to read without auto-scroll fighting you.
- **Charts and KPIs** — price history and summary stats tied to the active symbol, fed from PostgreSQL on the backend rather than recomputed in the browser.
- **Bot-activity panels** — ring gauges and hourly breakdowns that highlight automated vs organic flow (useful on thin markets and during listing pumps).
- **CSV export** — download visible stats for offline analysis or sharing in a desk channel.
- **Visitor analytics** — lightweight script tracks page engagement without blocking the WebSocket feed.

The browser is display-only: PostgreSQL writes, snapshot generation, and sitemap updates all happen server-side so a thousand concurrent readers do not hammer your database.

## Per-coin SEO pages

Every tracked symbol gets a canonical URL at **[/mexc/{SYMBOL}/](https://logicencoder.com/mexc/)** — for example `/mexc/BTCUSDT/` or `/mexc/PIUSDT/`. These are not static marketing stubs; Node SSR fills each page with **live trading data** at request time.

A typical coin page includes:

- **KPI strip** — current price, 24h change, volume, and headline metrics above the fold.
- **24h hourly table** — hour-by-hour price and volume so trends are visible without opening the main app.
- **Bot-activity ring** — visual share of flow that looks automated vs manual over the recent window.
- **Schema.org JSON-LD** — a `Dataset` block with 50+ fields (price, volume, buy/sell split, VWAP, utilization, symbol labels) so Google and AI crawlers index real numbers, not placeholder copy.
- **Chart PNG** — optional social preview image generated by the backend snapshot pipeline.

When SSR is unavailable, static snapshot HTML under `/snapshots/mexc/` still gives crawlers readable content. The WordPress plugin proxies SSR requests and merges backend pushes so operators never edit PHP templates per coin.

## Visitor experience

Public pages connect over a **WebSocket** feed using **MessagePack** frames — smaller on the wire than JSON at 1,400-pair scale. Visitors see:

- **Live tape** with color-coded sides, timestamps, and notional in USDT.
- **Scroll / freeze** — leave auto-scroll on for monitoring; freeze when you are reading a specific burst of prints.
- **Pair switch** — search or click a new row; the UI resubscribes without a full page reload.
- **Bot-activity context** — panels that summarize whether recent volume looks mechanical, alongside the raw tape.

Latency and compression are tuned for wide subscription lists: one backend process ingests MEXC once and broadcasts to all readers instead of each browser opening its own exchange socket.

## Monitor Dashboard

wp-admin **Monitor Dashboard** is the operator nerve center when something looks stale on the public site or trade counts drop after a deploy. Three panels mirror what you would otherwise SSH in to check.

### WebSocket throughput and health

Top status bar shows **API** and **SSR** reachability with response times, plus a green **Connected** badge when the admin monitor socket is live. Below that, a **WebSocket throughput** grid tracks:

- **Messages/sec** and **download KB/s** — is data actually flowing right now?
- **Peak rate** and **total downloaded** — cumulative volume since server start (hundreds of GB on a busy fleet).
- **Reconnects** and **server uptime** — spot flapping connections or recent restarts.
- **Min / avg / current latency** — end-to-end delay from exchange ingest to your browser (spikes here mean UI lag even if MEXC is fine).
- **Compression ratio** — MessagePack + gzip savings vs raw protobuf size.

A **3-hour rolling chart** plots download speed and messages/sec per minute so you can correlate a traffic drop with a deploy, a MEXC outage, or a subscription gap.

<img src="assets/websocket-monitor.png" alt="WebSocket monitor — throughput, latency, and API health" width="860" />

### Subscribed pairs and server metrics

The **subscribed pairs** chip wall lists every USDT market currently on the MEXC stream — typically 1,400+ blue tags in alphanumeric order. Below, four summary cards answer the questions operators ask first:

- **Subscribed pairs** — how many coins are on the server list vs how many are actively receiving trades (e.g. 1,460 live of 1,482 tracked). Footer shows aggregate **trades/min** and a green **MEXC connection healthy** check.
- **Broadcast queue** — depth of the internal fan-out queue (should stay near zero; a growing queue means clients or Postgres writes are falling behind).
- **PostgreSQL** — total trades stored server-side, messages sent to browsers, and server start time — proof the database leg is alive.
- **Connected clients** — unique IPs and WebSocket sessions reading the feed right now.

<img src="assets/server-metrics.png" alt="Subscribed pairs and server metrics — MEXC stream, queue, PostgreSQL, clients" width="860" />

### System logs

Dark **system log** console filtered to connection lifecycle — no per-trade spam. Typical boot sequence on one screen:

1. Config loaded from server.
2. Admin monitor initializes (MessagePack mode).
3. WebSocket connect attempt with URL and retry count.
4. Authentication success with masked API key prefix.
5. Bulk subscribe to **all** symbols (1,400+ pairs listed, truncated in UI).
6. Server welcome and **successfully subscribed** confirmation.

Use this panel when the public site shows zero trades/min but MEXC itself is up — you will see whether auth failed, subscribe timed out, or the socket never reconnect after a key rotation.

## Coin Manager

wp-admin **Coin Manager** is fleet hygiene for 1,400+ pairs — add new listings, prune delisted symbols, reload from the server bootstrap, and validate SEO output without leaving WordPress.

### Overview and dead coins

Four stat cards at the top summarize fleet health:

| Card | Meaning |
|------|---------|
| **Tracked coins** | Symbols on the server list; footer shows aggregate trades/min across the fleet. |
| **Live stream** | Pairs receiving trades right now; footer lists how many are idle (no print in 60s). |
| **Never traded** | Added but no historical print yet — often right after listing. |
| **Dead / missing** | Needs attention — delisted on MEXC, removed from stream, or never subscribed. |

The **Dead & missing** panel lists red chips for symbols no longer on the exchange (e.g. delisted meme tickers). One click removes them from the tracked set so sitemap and SSR do not 404 forever. **Refresh dead list** pulls the latest diff from the backend.

<img src="assets/coin-manager-overview.png" alt="Coin Manager overview — tracked, live, dead, and missing counts" width="860" />

### Add, reload, and tracked list

**Reload symbols on server** syncs the WordPress coin list with the Python bootstrap — run after a backend deploy or MEXC listing batch.

- **Single add** — type `BTCUSDT`, press Add coin.
- **Bulk import** — paste one symbol per line, **Add all** for dozens of new listings at once.

The **All tracked coins** table is the sortable fleet roster: symbol, **trades/min**, **stream** status (live vs idle), **ever traded** flag, and **Remove** per row. **Refresh now** updates stream stats without reloading the page — use it when you just added a coin and want to confirm it went live.

<img src="assets/coin-manager-tracked.png" alt="All tracked coins — trades per minute, stream status, remove action" width="860" />

### Schema.org validation

Before you push a new coin to production SEO, pick a symbol and hit **Run validation**. The tool fetches two layers:

**Python API — server stats** for the symbol: connected clients, messages sent, latency histogram (min / max / avg), compression ratio, and subscription room counts. Confirms the data pipe is warm before you judge HTML.

**Node SSR — HTML structure checks** for the same symbol:

- Bot Activity section present
- 24h hourly table present
- KPI strip present
- Ring gauge (bot) present
- Schema.org JSON-LD block present
- Symbol string in document
- HTML byte size (sanity check against empty fallback pages)

Below that, an expandable **JSON-LD key inventory** — every Schema.org field the SSR emitted (name, description, `current_price_usdt`, 24h high/low, buy/sell volume split, VWAP, percentage changes, and dozens more). Operators use this to confirm Google Rich Results will see real `Dataset` fields, not a stub.

## Sitemap and IndexNow

Search discovery is first-class — 1,400+ coin URLs would be impossible to submit by hand.

### Sitemap status and IndexNow queue

**MEXC Sitemap & Indexing** dashboard shows:

- **Public sitemap URL** (e.g. `mexc-coins-sitemap.xml`) with live HTTP check — confirms WordPress serves a valid XML urlset.
- **WordPress hook** registered and **static file** present in the plugin for fallback.
- **Total URLs** in sitemap (~1,479) and cache timestamp.
- Buttons to **view XML**, open **Google Search Console**, and **save static sitemap file** after bulk coin changes.

**IndexNow queue** panel tracks batch submissions to Bing/Yandex:

- Pending URLs, last batch time/status, sent vs remaining counts.
- **Last push new coins** — how many fresh `/mexc/SYMBOL/` URLs went out after the last listing batch.

<img src="assets/sitemap-indexing.png" alt="MEXC Sitemap and IndexNow queue status" width="860" />

### Auto-push, coin table, and API sync

- **Auto push on publish** — when a WordPress post or page goes live, its permalink is sent to IndexNow automatically (status and last URL shown).
- **Coins in sitemap** — paginated table of symbol + `/mexc/SYMBOL/` path so you can spot a typo before Google does.
- **Sync now** — pulls the authoritative coin list from the Python API bootstrap and rebuilds sitemap entries. Run after Coin Manager bulk import or backend listing update.
